# v0.5.0 Control API Roadmap

Work in small vertical slices. Each slice should leave Telegram behavior intact
unless it explicitly changes the Telegram adapter contract.

## Slice 1: Architecture Docs

- Add ADR-019 and control-plane research docs.
- Reposition README and wiki.
- Update contract and regression docs.
- No runtime code changes.

## Slice 2: Control Interface Extraction

- Introduce adapter-independent Go interfaces for threads, turns, events,
  approvals, skills, and notifications.
- Keep the existing App Server client as the first implementation.
- Do not add HTTP yet.

## Slice 3: App Server Capability Map

- Add a schema-backed capability map for installed Codex versions.
- Detect support for thread fork, skills, hooks, MCP status, app list, review,
  command exec, and future goal APIs.
- Tests must cover missing capabilities and docs/schema drift.

## Slice 4: Adapter-Independent Events

- Move normalized lifecycle/tool/final/approval/input events out of
  Telegram-specific rendering paths.
- Preserve ADR-013: no session JSONL live overlay.
- Telegram should consume the new event model without visible regressions.

## Slice 5: Telegram Adapter On Control API

- Keep current Telegram commands, Details, notifications, and Plan behavior.
- Route Telegram actions through the control interface.
- Live Telegram E2E remains required for user-visible Telegram changes.

## Slice 6: Router-Agent Prototype API

- Add a local-only experimental API for a router-agent consumer.
- Prefer loopback or unix socket.
- Initial operations: list/search/read threads, start/steer/interrupt turns,
  subscribe events, approve/answer input, list skills.
- Keep auth and exposure conservative; no public network listener.

## Slice 7: Voice Adapter Research/Prototype

- Prototype wake word and chained voice pipeline outside the core first.
- Keep approval and notification policy in the control core.
- Do not bypass Codex permissions, approvals, or sandbox constraints.

## Release Gate

v0.5.0 should ship when:

- Telegram adapter remains compatible with v0.4 behavior;
- control interfaces are covered by unit tests;
- App Server capability map handles missing/drifting methods;
- docs and ADRs match the implemented boundaries.
