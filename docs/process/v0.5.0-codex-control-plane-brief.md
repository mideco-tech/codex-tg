# v0.5.0 Codex Control Plane Brief

## Problem

Official Codex Remote Connections now cover the basic "use Codex from a phone"
story. Keeping `codex-tg` positioned only as a Telegram remote UI makes the
project overlap with the official product instead of building on top of it.

The durable value in the project is the local Codex control loop: thread
identity, turn lifecycle, approvals, live events, notifications, logs, and
operator routing. That layer should become reusable outside Telegram.

## Goal

Turn `codex-tg` toward a local Codex Control Plane:

- expose an adapter-independent control core for Codex threads, turns, events,
  approvals, skills, and notifications;
- keep Telegram as the first adapter, not the product center;
- make future router-agent and voice-assistant work consume the same control
  layer instead of duplicating Telegram-specific logic.

## Non-goals

- No runtime code changes in the architecture documentation slice.
- No public cloud service.
- No repo rename.
- No voice implementation.
- No replacement of official Remote Connections.
- No replacement of App Server as the authoritative interactive Codex state
  source.

## UX / Operator Flow

The current v0.4 Telegram flow remains supported. In the v0.5 direction,
Telegram becomes one channel adapter for a broader local control surface:

```text
User intent
  -> router agent or channel adapter
  -> codex-control core
  -> Codex App Server / orchestration adapters
  -> normalized events and notification policy
  -> Telegram, tray, voice, or future UI adapter
```

## Domain Model

- **Control Core**: adapter-independent service for Codex operations.
- **Codex Connector**: App Server-backed implementation of thread, turn, event,
  approval, skill, and ecosystem operations.
- **Channel Adapter**: Telegram, tray, voice, local HTTP, or future mobile UI.
- **Router Agent**: future external consumer that decides which Codex action to
  perform from user intent.
- **Notification Policy**: severity mapping such as `urgent`, `normal`,
  `silent`, and `digest`.

## Architecture

App Server remains the primary control surface for interactive Codex state.
SDK/MCP integrations may be used as orchestration adapters when ADR-019 allows
them, especially for Agents SDK workflows, handoffs, traces, and multi-agent
experiments.

The current spawned stdio App Server transport remains supported. v0.5 work may
prepare for official App Server `unix://` and `app-server proxy` transports, but
must keep App Server events and snapshots as the live source of truth.

## Testing

Docs-only architecture work requires:

- `git diff --check`;
- targeted secret/local scan over edited docs;
- link/path sanity check for new documentation links.

Later implementation slices must add tests for control-core interfaces, schema
capability detection, event normalization, notification severity, and Telegram
adapter compatibility.

- [ ] README and wiki describe Telegram as an adapter.
- [ ] Research docs record official and local evidence without private data.
- [ ] Contract matrix includes the Codex Control API contract.
- [ ] Regression map includes control-plane test anchors.
- [ ] AGENTS.md points future agents at ADR-019 for SDK/MCP decisions.
