# Feature Brief Template

Use this for new or ambiguous feature work. Keep it short and delete or archive stale briefs after durable decisions move into ADRs, contract docs, or tests.

## Problem

Who has the problem, and what fails today?

## Goal

What must be true after this change?

## Non-goals

What are we explicitly not changing in this slice?

## UX / Operator Flow

Describe the user-visible path, including Telegram messages, buttons, callbacks, and failure states when relevant.

## Domain Model

List new or changed concepts, states, ids, routes, callbacks, or storage records.

## Architecture

Name the module boundary and the public interface shape. Prefer deep modules with small interfaces.

## Testing

List unit, integration, regression, and live E2E checks.
