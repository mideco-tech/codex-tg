# v0.2.0 Live App Server Events Brief

## Summary

`v0.2.0` moves Telegram-origin run rendering from polling-first visibility to App Server live event visibility. The daemon still treats `thread/read` as the durable reconciliation source, but live App Server `item/*` notifications may render the current command for turns that were started from Telegram and are marked as Telegram-origin.

The release is implemented as slice commits. Each slice must pass targeted tests and the relevant live Telegram E2E case before the next slice starts.

## Target Behavior

- Telegram-origin active turns show the current command in `[Tool]` when App Server live events provide a matching `thread_id + turn_id`.
- Telegram-origin completed tools return to the existing `Last completed tool:` and `Last completed output:` presentation.
- Foreign GUI/CLI observer runs keep the `v0.1.3` contract: `[Tool]` shows the last completed tool, or neutral absence.
- Session JSONL stays out of live Telegram UI. It may only be used by explicit export/full-log paths.
- Missing, null, empty, or literal `"<nil>"` fields remain absent values and must not render into Telegram.

## Implementation Cycle

1. Write or update the failing tests for the current slice.
2. Implement the smallest behavior change.
3. Run targeted Go tests for the touched packages.
4. Rebuild/restart the daemon when Telegram-facing behavior changed.
5. Run the targeted live Telegram E2E case.
6. Commit the slice only after tests and live validation pass, or record the exact blocker.

## Out Of Scope

- No cloud relay, account server, mobile app, or multi-machine protocol.
- No App Server protocol changes.
- No broker process that survives Telegram daemon restart.
- No reattach/resume of an already-running App Server turn after daemon death.
- No new operator config flag by default.
